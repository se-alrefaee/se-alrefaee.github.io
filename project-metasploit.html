Project overview

This lab simulates a full penetration test against Windows systems using the Metasploit framework. The goal is to understand how attackers deliver payloads, obtain a Meterpreter session, and perform post‑exploitation actions so that these techniques can be detected and mitigated in real environments.

Objectives

  • Set up an isolated lab with Kali Linux and Windows targets for safe exploitation practice.
  • Generate and deliver custom Metasploit payloads to establish reverse shells on victim machines.
  • Explore Meterpreter post‑exploitation modules such as keylogging, screenshot capture, and file manipulation.
  • Relate every offensive technique to the corresponding defensive controls and monitoring strategies.

Technical implementation

Lab setup. I created multiple virtual machines, including Kali Linux as the attacker, Windows Server 2022 as a domain‑like environment, and a Windows client, all connected to an internal network segment.

Payload creation. Using msfvenom, I generated Windows reverse shell payloads and configured listeners in Metasploit to receive Meterpreter sessions when the payload executed.

Initial access. I simulated social engineering by delivering the payload as an attachment or executable that the user might be tricked into running, reproducing a common real‑world attack path.

Post‑exploitation. Once a Meterpreter session was established, I tested commands for system information gathering, privilege escalation attempts, keylogging, and screenshot capture to understand the attacker’s visibility.

Defensive perspective. At each step, I noted which logs, alerts, and security tools could detect the activity, such as antivirus, EDR, or Windows event logs.

Challenges and difficulties

  • Antivirus and security controls. A major difficulty was that modern Windows systems and built‑in antivirus quickly detected and deleted basic payloads. I had to experiment with different encoders, payload formats, and execution methods to get a stable session in a realistic way.
  • Network and routing configuration. At the beginning, reverse shells failed to connect because of incorrect IP addressing and NAT rules between the virtual machines. Troubleshooting routing, firewall settings, and listener configuration took time, but it helped me better understand how Metasploit communicates over the network.

How I organized the work

I split the project into phases: lab setup, payload generation, initial exploitation, post‑exploitation, and documentation. For each phase, I maintained a checklist in a simple Kanban board and documented every successful command in a lab notebook for reproducibility.

I also used GitHub to store configuration notes and important commands, which made it easy to roll back failed ideas and keep track of what worked for each Windows version.

Key learnings

  • Metasploit is extremely powerful, but reliable exploitation still requires solid understanding of networking and operating system behavior.
  • Even simple payloads can be visible in logs and security tools, which shows the importance of centralized logging and endpoint monitoring.
  • Hands‑on lab work is the best way to understand both the limitations and the real risks of common exploitation frameworks.

Core features practiced

  • Creation and configuration of reverse TCP shell payloads.
  • Launching and managing multiple Meterpreter sessions.
  • Running post‑exploitation modules: keylogger, screenshot, file upload/download.
  • Basic privilege escalation attempts and system reconnaissance.

Tools and technologies

  • Metasploit Framework and msfconsole.
  • msfvenom for payload generation and encoding.
  • Kali Linux as the main attack platform.
  • Windows Server 2022 and Windows 10/11 as targets.
  • VirtualBox/Proxmox for isolated lab networks.

Defensive takeaway

Observing the full attack chain with Metasploit helps to design logging, detection rules, and hardening measures that target real adversary behavior instead of only theoretical scenarios.
