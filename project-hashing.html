Project overview

This project focuses on cryptographic hashing as a way to verify data integrity and detect tampering. Through experiments with several algorithms, I observed how small changes in a file completely modify the hash and learned why some older algorithms are no longer considered secure.

Objectives

  • Understand what properties a good cryptographic hash function must provide.
  • Use hashes to detect file corruption and unauthorized modifications.
  • Compare different algorithms, such as MD5 and SHA‑256, and discuss their strengths and weaknesses.
  • Relate hashing to real‑world use cases like password storage and software distribution.

Technical implementation

File hashing. I generated hashes for various files, saved them, then modified the files slightly (for example, by changing one character) and re‑generated the hashes to observe the avalanche effect.

Integrity checks. I simulated file transmission over an untrusted channel and verified that comparing hashes before and after transfer quickly reveals any corruption or tampering.

Password storage concept. I studied how systems store hashed passwords instead of clear text, and how during login only the hash of the input is compared to the stored value.

Algorithm comparison. I contrasted MD5 and SHA‑256, noting that MD5 is considered broken for security‑critical use, while SHA‑256 is still widely used for integrity verification.

Challenges and difficulties

  • Understanding collision risks. At first, it was not obvious why collisions are such a serious problem if hashes look random. I had to read more examples and think about how an attacker could abuse collisions to replace files or certificates.
  • Separating hashing from encryption. It was easy to confuse hashing with encryption, especially when looking at password storage. Writing down clear definitions and use cases helped me keep these concepts separate.

How I organized the work

I structured the lab in three parts: theory recap, practical experiments, and real‑world examples. For each algorithm, I documented the command used, sample inputs, outputs, and short comments about what I observed.

I also created a small table in my notes comparing algorithms (output length, speed, known weaknesses), which made it easier to explain why MD5 should be avoided in modern systems.

Key learnings

  • Good hash functions make it extremely hard to find two different inputs with the same output.
  • Even a single bit change in the input produces a completely different hash, which is ideal for integrity checks.
  • Older algorithms like MD5 can still be useful for non‑security‑critical tasks, but stronger options like SHA‑256 are preferred in security contexts.

Core activities

  • Generating hashes for text and files with different algorithms.
  • Simulating data transmission and verifying integrity with hash comparisons.
  • Discussing password hashing and the importance of not storing plaintext credentials.

Tools and technologies

  • Command‑line hashing utilities (Linux).
  • Text editors for modifying files and observing effects.
  • Spreadsheets or notes to summarize algorithm properties.

Practical impact

Understanding hashing at a practical level clarified how file checksums, package managers, and password databases use the same fundamental concept to protect data integrity and user accounts.
